# Руководство по внесению изменений (Contributing Guide)

Мы рады, что вы хотите помочь в разработке проекта **MyPortfolio**! Пожалуйста, следуйте этому руководству, чтобы процесс настройки и разработки был быстрым и удобным.

---

## 🛠️ Требования (Prerequisites)

Перед началом работы убедитесь, что у вас установлены:
- **Node.js** (версия 20.x или выше)
- **NPM** (версия 10.x или выше)
- **Docker** и **Docker Compose** (для запуска локальной базы данных PostgreSQL и Nginx в продакшене)

---

## 🚀 Быстрый запуск разработки (Development Setup)

### 1. Настройка окружения
Проект разделен на две части: `backend` и `frontend`. В корнях каждой из папок необходимо настроить файлы конфигурации:
- В `/backend` скопируйте `.env.example` в `.env.dev` (для локальной разработки) и настройте параметры подключения к БД.
- В `/frontend` проверьте настройки в `.env` (если требуются кастомные эндпоинты).

### 2. Запуск локальной базы данных
Самый простой способ поднять базу данных PostgreSQL — использовать Docker:
```bash
# Из корня проекта запускаем только контейнер базы данных
docker-compose up -d db
```
База данных будет доступна локально на порту `5433` (или `5432` в зависимости от конфигурации в `docker-compose.yml`).

### 3. Запуск Backend (NestJS)
Перейдите в папку бэкенда, установите зависимости и запустите сервер в режиме разработки:
```bash
cd backend
npm install
npm run start:dev
```
Сервер запустится на [http://localhost:3001](http://localhost:3001), а Swagger-документация API будет доступна по адресу [http://localhost:3001/docs](http://localhost:3001/docs).

### 4. Запуск Frontend (React + Vite)
Перейдите в папку фронтенда, установите зависимости и запустите локальный сервер разработки:
```bash
cd ../frontend
npm install
npm run dev
```
Фронтенд будет доступен по адресу [http://localhost:5173](http://localhost:5173).

## 🗄️ Работа с миграциями (Migrations)

Все изменения в схеме базы данных должны проводиться через миграции TypeORM. Не используйте автоматическую синхронизацию в продакшене.

### Создание новой миграции:
1. Внесите изменения в соответствующий класс сущности в `backend/src/admin/entities/`.
2. Перейдите в каталог `backend` и сгенерируйте файл миграции:
   ```bash
   npm run migration:generate -- src/migrations/NameOfYourMigration
   ```
3. Проверьте сгенерированный файл в папке `src/migrations/`, чтобы убедиться, что изменения соответствуют ожидаемым.

### Применение миграции:
```bash
npm run migration:run
```

### Откат последней миграции:
```bash
npm run migration:revert
```

## 🧪 Тестирование (Testing)

Перед отправкой изменений (Pull Request) убедитесь, что все тесты выполняются без ошибок.

### Тестирование бэкенда (Backend Tests)
- **Unit-тесты**:
  ```bash
  cd backend
  npm run test
  ```
- **E2E-тесты** (требуют запущенной тестовой базы данных):
  ```bash
  npm run test:e2e # или npx jest --config test/jest-e2e.config.js
  ```

### Тестирование фронтенда (Frontend Tests)
- **Юнит-тесты компонентов (Vitest)**:
  ```bash
  cd frontend
  npm run test:run
  ```

## 📝 Стандарты оформления кода (Style Guide)

### Форматирование
- Мы используем **Prettier** и **ESLint** для соблюдения единого стиля.
- Вы можете запустить автоисправление:
  - Бэкенд: `npm run lint` (из каталога `backend`)
  - Фронтенд: `npm run lint` (из каталога `frontend`)

### Правила оформления коммитов (Commit Message Convention)
Мы следуем спецификации [Conventional Commits](https://www.conventionalcommits.org/). Сообщения коммитов должны иметь формат:
`тип(область): описание`

Примеры типов:
- `feat` — добавление нового функционала (например: `feat(auth): add email verification`)
- `fix` — исправление багов (например: `fix(stats): prevent double visit tracking`)
- `docs` — обновление документации (например: `docs: update deployment guide`)
- `style` — правки стиля, форматирования (без изменения бизнес-логики)
- `refactor` — рефакторинг кода (например: `refactor(user): simplify user roles check`)
- `test` — добавление или изменение тестов

## 📥 Отправка изменений (Submitting a PR)

1. Создайте свою ветку от `main` (например, `feature/add-pwa-support` или `bugfix/fix-captcha`).
2. Внесите изменения, соблюдая стандарты и написав необходимые тесты.
3. Убедитесь, что CI/CD пайплайн (GitHub Actions) проходит успешно.
4. Откройте Pull Request и опишите проделанную работу.
